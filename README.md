# ThoughtCurrent

An MCP server that compiles text from multiple SaaS sources into local markdown files. Runs as a global Claude Code tool — available in every project, every session, with zero manual server management.

## What It Does

ThoughtCurrent is a **read-only data pipe**. It fetches from your tools, normalizes everything into markdown, and writes it locally so Claude (or you) can read it as context for any project.

**Supported sources:**

| Source | What It Fetches | Auth |
|--------|----------------|------|
| **Slack** | Messages, files, canvases, images, pins, bookmarks (with thread replies + text extraction) | User or Bot token |
| **GitHub** | Issues, PRs, and comments | `gh` CLI (local auth) |
| **Gmail** | Emails and attachments | OAuth 2.0 |
| **Linear** | Issues with comments | API key |
| **Granola** | Meeting transcripts | Local app token (auto-detected) |
| **Trello** | Cards, checklists, comments | OAuth |
| **Figma** | Text layers, comments, screenshots | Access token |
| **Sentry** | Issues with stack traces, event metadata | Bearer token |
| **Datadog** | Error logs (500s, configurable queries) | API key + App key |
| **Hugging Face** | Inference endpoint status and errors | API token |

## How It Works

ThoughtCurrent runs as an MCP server over stdio. Claude Code spawns it on first tool call and manages the lifecycle automatically.

```
You: "compile messenger-recent"
Claude: calls compile("messenger-recent") MCP tool
        → ThoughtCurrent fetches from Slack, GitHub, Linear...
        → writes markdown to output/messenger-recent/
        → returns progress + item counts
Claude: reads the compiled output and uses it as context
```

No manual server management. No web UI. Just tools.

## Setup

### 1. Clone and install

```bash
git clone <repo-url> ~/work/ThoughtCurrent
cd ~/work/ThoughtCurrent
bun install
```

### 2. Configure credentials

Create a `.env` file in the project root:

```bash
# Slack (user token preferred — sees DMs, private channels, etc.)
SLACK_USER_TOKEN=xoxp-...
# OR
SLACK_BOT_TOKEN=xoxb-...

# GitHub (uses gh CLI auth — just run `gh auth login` separately)
# No env var needed — repos are configured per-preset (e.g. ["org/repo"])

# Gmail (OAuth — run auth script in step 3)
GOOGLE_CLIENT_ID=your-client-id
GOOGLE_CLIENT_SECRET=your-client-secret
# GOOGLE_REFRESH_TOKEN=  ← auto-generated by auth script

# Linear
LINEAR_API_KEY=lin_api_...

# Granola (no env var needed — reads token from the Granola desktop app)

# Trello (OAuth — run auth script in step 3)
TRELLO_API_KEY=your-api-key
# TRELLO_TOKEN=  ← auto-generated by auth script

# Figma
FIGMA_ACCESS_TOKEN=figd_...

# Sentry
SENTRY_AUTH_TOKEN=sntrys_...
SENTRY_ORG=your-org-slug

# Datadog
DATADOG_API_KEY=your-api-key
DATADOG_APP_KEY=your-app-key
# DATADOG_SITE=datadoghq.com  # default; change for EU/other sites

# Hugging Face
HF_TOKEN=hf_...
# HF_NAMESPACE=BuiltByHQ  # default; your HF organization/user
```

You only need tokens for the sources you plan to use. Everything else will show as `not_configured` in health checks.

### 3. OAuth setup (Gmail & Trello)

For OAuth sources, run the standalone auth scripts. They spin up a temporary HTTP server on port 3141, open your browser for consent, handle the callback, and save the token to `.env`.

```bash
# Gmail
bun run scripts/auth-gmail.ts

# Trello
bun run scripts/auth-trello.ts
```

**Prerequisites for Gmail OAuth:** You need a Google Cloud project with the Gmail API enabled and OAuth credentials configured with `http://localhost:3141/api/auth/gmail/callback` as an authorized redirect URI.

**Prerequisites for Trello OAuth:** You need a Trello Power-Up API key from https://trello.com/power-ups/admin.

### 4. Register with Claude Code

Add to your global Claude Code settings (`~/.claude/settings.json`):

```json
{
  "mcpServers": {
    "thoughtcurrent": {
      "command": "bun",
      "args": ["run", "/absolute/path/to/ThoughtCurrent/src/index.ts"]
    }
  }
}
```

Start a new Claude Code session — the tools are now available globally.

## Usage

### Presets

Presets are named filter configurations that define which sources to pull from, with what filters. Each preset gets its own output directory, cache, and error logs.

**Create a preset** (conversationally via Claude):
```
"Create a ThoughtCurrent preset called 'my-project' that pulls from:
- Slack channels: team-backend, deploys (last 2 weeks)
- GitHub repo: org/my-project
- Linear with keyword 'ENG'"
```

**Compile a preset:**
```
"Compile my-project"
```

**Multiple presets per project** are encouraged. Use different presets for different scopes:
- `my-project-recent` — last 2 weeks, all sources
- `my-project-auth-bug` — last 3 months, keywords: ["auth", "session", "token"]
- `my-project-q1` — Jan-Mar date range, planning channels only

### What Can You Do With It?

Once a preset is configured, just talk naturally from any project. Claude compiles the data, reads the output, and answers with cross-tool context you'd normally have to tab between 7 apps to assemble.

**Morning catch-up:**
```
"Compile my-project and catch me up"
→ Summary of new PRs, Linear issues, Slack threads, Sentry errors, Trello card movements
```

**Bug investigation:**
```
"Users are seeing a null error on the editor page — what do we know?"
→ Cross-references Sentry stack traces with Slack discussions and Linear issues
```

**PR writing with business context:**
```
"I just finished the checklist feature, write the PR description"
→ Pulls context from the Linear ticket, Slack design discussions, and Trello card
```

**Sprint planning:**
```
"What should we prioritize this week?"
→ Top Sentry errors by frequency, open Linear issues, active Trello cards, unresolved Slack threads
```

**Pre-meeting prep:**
```
"I have a sync in 10 minutes, what do I need to know?"
→ Last meeting's Granola transcript + what's changed since in Slack, Linear, and GitHub
```

**Code review with full context:**
```
"Review PR #47 on org/my-project"
→ PR diff alongside the Linear ticket, related Slack discussion, and Sentry errors in that code path
```

**Model/infra health check:**
```
"Is our ML endpoint healthy?"
→ HuggingFace endpoint status — running, scaled down, or erroring
```

### MCP Tools

| Tool | Args | Description |
|------|------|-------------|
| `compile` | `preset` | Start async compilation. Returns a job ID. |
| `check_compilation` | `jobId` | Poll compilation status. Returns completed/failed/pending sources. |
| `check_status` | — | Health check all configured sources. |
| `list_presets` | — | List all saved presets with their full configs. |
| `save_preset` | `name`, `sourceConfigs` | Create or update a preset. |
| `update_preset` | `name`, `sourceConfigs` | Update an existing preset's config. |
| `delete_preset` | `name` | Delete a preset. |
| `clear_output` | `preset`, `source?` | Clear output for a preset. Optionally clear only one source. |
| `list_slack_channels` | — | List Slack channels (for building presets interactively). |
| `list_slack_users` | — | List Slack users (for building presets interactively). |

### Compilation Flow

1. `compile("my-project")` → returns `{ jobId, status: "running" }`
2. Poll `check_compilation(jobId)` → `{ status: "running", completed: ["slack"], pending: ["github"] }`
3. Poll again → `{ status: "completed", completed: ["slack", "github"], failed: [], totalItems: 142, outputPath: "..." }`
4. Claude reads the compiled markdown from the output path

Partial failures are handled gracefully — if Slack compiles but GitHub errors, you still get the Slack output.

## Output Structure

Output is namespaced by preset name:

```
output/
  my-project/
    _compiled.md              # Chronological merge of ALL sources
    slack/
      _index.md               # TOC with counts
      messages/
        by-channel/           # One .md file per channel
        by-user/              # Grouped by author
        by-date/              # Grouped by date
      files/
        documents/
        images/
        canvases/
      pins/
      bookmarks/
    github/
      issues-and-prs.md
    linear/
      issues.md
    gmail/
      _index.md
      by-date/
      by-sender/
    granola/
      _index.md
      by-date/
      by-meeting/
    trello/
      cards.md
    sentry/
      errors.md
    datadog/
      logs.md
    huggingface/
      endpoints.md
    figma/
      _index.md
      text/
      comments/
      screenshots/
    .meta/
      cache.json              # Dedup cache — incremental compilations skip cached items
    .logs/
      <jobId>.log             # Debug log per compilation run (keeps last 10)
    .errors/
      <source>_<date>.md      # Persistent error report (auto-clears on next success)
```

## Logging & Debugging

Three logging layers, from broad to specific:

| Layer | Location | Lifecycle |
|-------|----------|-----------|
| MCP server log | `.logs/mcp.log` | Always on, server-level events |
| Compilation logs | `output/<preset>/.logs/<jobId>.log` | Per-run: every API call, cache hit/miss, timing |
| Error reports | `output/<preset>/.errors/<source>_<date>.md` | Persists until source compiles successfully |

Error reports include:
- Full error message and stack trace
- The filter config that triggered it
- Suggested fix (e.g., "Token expired — re-run auth script")

## Architecture

```
┌─────────────────────────────────────┐
│         Claude Code Session         │
│  (any project, any conversation)    │
└──────────────┬──────────────────────┘
               │ MCP (stdio)
┌──────────────▼──────────────────────┐
│      ThoughtCurrent MCP Server      │
│                                     │
│  Tools → Async Job Manager          │
│              │                      │
│       Compilation Pipeline          │
│         (compile.ts + cache.ts)     │
│              │                      │
│       Source Fetchers (read-only)    │
│  slack│github│gmail│linear│...      │
│              │                      │
│       Writer + Logger               │
│  (markdown output + debug logs)     │
└──────────────┬──────────────────────┘
               │
          output/<preset>/
```

## Key Design Decisions

- **Read-only** — never writes to external systems. All API calls are GET/read-scoped.
- **Preset-based** — no per-project config files. Presets are centralized, reusable, and composable.
- **Async compilation** — `compile` returns immediately with a job ID. Poll with `check_compilation`.
- **Incremental** — cache stores external IDs per preset. Re-running only fetches new items since last compile.
- **Preset-isolated output** — each preset has its own output directory, cache, logs, and error reports.
- **Error persistence** — failed sources write detailed error reports that stay until resolved.
- **No web UI** — Claude Code is the sole interface. The MCP server is a data pipe.
- **stdio transport** — Claude Code manages the process lifecycle. No ports, no daemons.

## File Extraction

ThoughtCurrent extracts text from files found in Slack and Gmail attachments:

| Format | Library |
|--------|---------|
| PDF | pdf-parse |
| Word (.docx, .doc) | mammoth |
| Excel (.xlsx, .xls, .csv) | exceljs |
| PowerPoint (.pptx) | fflate (ZIP extraction) |
| HTML | Built-in tag stripping |
| ZIP archives | Recursive extraction |
| Video | Screenshot extraction via ffmpeg (if installed) |
